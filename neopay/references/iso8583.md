## HESO93 Primary Bitmap

HESO93 uses a 64-field primary bitmap (fields 1–64). Field 1 is always the secondary bitmap (present when any field 65–128 is used).

```
Bit position → Field number
1 = field 2, 2 = field 3, ... 64 = field 65
```

**Bitmap construction (binary/ASCII variants):**
- **Binary:** 8 bytes, each bit set = field present (0x80 = field 1, 0x40 = field 2, etc.)
- **ASCII hex:** 16 bytes, `0`–`F` hex chars per nibble

## Variable-Length Fields (LL/LLL)

Fields with `LL..` prefix encode their own length:
- `LL..`: 2-digit length prefix (max 99 bytes)
- `LLL..`: 3-digit length prefix (max 999 bytes — used in field 48, 55)

**Example — Field 2 (PAN):**
```
Input PAN: "4111111111111111"
Encoding: "16" + "4111111111111111" = "164111111111111111"
Length bytes = 2 chars
```

**Example — Field 48 (Extended Data, 3-digit prefix):**
```
Data: {"terminalId":"TERM001","merchantId":"MERCH001"}
JSON length: 54 chars
Encoding: "054" + JSON string (3-digit prefix)
```

## Binary vs ASCII Dialects

| Aspect | Binary (HISO93) | ASCII (HISO87) |
|--------|-----------------|----------------|
| PAN | BCD nibbles | ASCII digits |
| Amount | BCD (6 or 12 digits) | ASCII N |
| Bitmap | 8 bytes (raw bits) | 16 bytes hex chars |
| MAC/PIN | Raw 8 bytes | 16 bytes hex |
| Dates | BCD packed | ASCII N |
| Variable fields | BCD length + raw data | ASCII length + ASCII data |

## Processing Code (Field 3) — HISO93

```
NNN N N N
│││ │ │ │
│││ │ │ └── To Account type
│││ │ └── From Account type
│││ └── Transaction Code
││└── Product code
└── Function code
```

**Common Transaction Codes:**
- `000000` = goods/services purchase
- `200000` = cash advance
- `090000` = refund/credit
- `010000` = balance inquiry

**Account Types:**
- `00` = default/specified by card type
- `20` = savings account
- `30` = checking account
- `40` = credit card account
