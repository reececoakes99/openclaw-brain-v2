# Token Vault

## Overview

The token vault replaces sensitive PAN (Primary Account Number) data with secure tokens, eliminating the need to store raw card data. This is the primary mechanism for PCI DSS scope reduction.

## Tokenization Schemes

### Vault Tokens (Internal)
- Generated by Paybox token vault
- Format: `pb_tok_xxxxxxxxxxxxxxxxxxxxxxxx` (26 chars)
- Reversible: can detokenize to PAN using HSM-protected key
- Scope: single merchant / single Paybox installation

### Network Tokens (Scheme-Backed)
- Generated by Visa Token Service (VTS) or Mastercard Tokenization Platform
- Format: `visatok_xxxxxxxxxxx` / `mctok_xxxxxxxxxxx`
- Irreversible: network holds mapping, Paybox cannot detokenize
- Benefits: token updated automatically on card update/cancel, better authorization rates

**Preferred flow:** Card-on-file → network token → scheme handles updates

## Token Lifecycle

### Creation
```
1. Merchant sends PAN (via HTTPS directly to tokenization endpoint, never logged)
2. HSM: generate random token_id
3. HSM: encrypt PAN under DEK (Data Encryption Key)
4. Store: token_id → encrypted_PAN + PAN last4 + expiry + card_network
5. Return token to merchant (PAN never leaves HSM boundary)
```

### Update (Card Re-issue)
- Network token: VTS/MC platform updates automatically, push notification to Paybox
- Vault token: merchant calls update endpoint with new PAN → re-tokenize

### Expiry / Deletion
- Tokens expire when card expires (checked on every transaction)
- Merchant can delete token (for guest checkout removal, GDPR right to erasure)
- Deletion is soft-delete: audit trail retained for 7 years

## Key Hierarchy

```
LMK (Local Master Key) — stored in HSM, never exported
  └── KEK1 (Key Encryption Key 1)
        └── DEK1 (Data Encryption Key 1) — AES-256-GCM, rotates annually
              └── PAN (encrypted)
```

- KEK is generated by HSM, never leaves HSM
- DEK encrypted under KEK, stored in database (never in plaintext)
- Each merchant can have dedicated DEK (enhanced isolation)
- Key rotation: re-encrypt all PANs with new DEK, old DEK retained for decryption

## PAN Tokenization (AES-256-GCM)

```python
# Encryption flow
token_id = os.urandom(16).hex()
nonce = os.urandom(12)  # GCM nonce
dek = get_dek_from_hsm()  # 32 bytes

cipher = AESGCM(dek)
ciphertext = cipher.encrypt(nonce, pan.encode(), associated_data=token_id.encode())
# ciphertext = nonce (12) + tag (16) + encrypted_pan
store_token(token_id, ciphertext)
```

## Network Token Lifecycle (Visa/MC)

### Visa Token Service
1. POST `/v1/tokens` with PAN + VTSA credentials
2. VTS returns `vtas_token` + `cryptogram`
3. Authorization: include cryptogram in ISO8583 DE55
4. VTS updates: on card update, VTS notifies via webhook (`token.updated`)

### Mastercard Tokenization
1. MDES (MDES Token Service) API call with PAN
2. MDES returns `token` + `dynamic cryptogram`
3. On card update: MDES webhook notification

## PCI DSS Exemption Criteria

Tokenized PANs (where the token cannot be reverse-engineered to the PAN) are **not subject to PCI DSS storage requirements** for the PAN itself. However:

- Token vault itself must be PCI DSS compliant (HW/SW, network controls)
- Last4 PAN is considered masked PAN — still in scope
- Expiry date, cardholder name — in scope
- CVV/CVV2 — never stored, never in scope

**Tokenization architecture qualifies for SAQ A** (simplest PCI form) for merchants if:
- Token received via form direct-post (merchant never touches PAN)
- Merchant only stores token, never has access to raw PAN

## Detokenization API

```bash
# Detokenize (requires PCI DSS compliant environment + audit logging)
POST /v1/tokens/detokenize
Authorization: Bearer $SECRET_KEY
X-Token-Idempotency-Key: $IDEMPOTENCY_KEY

{
  "token": "pb_tok_xxxxxxxxxxxxxxxxxxxxxxxx"
}

# Response
{
  "pan": "4242424242424242",
  "expiry_month": 12,
  "expiry_year": 2027,
  "card_type": "visa"
}
```

**Security controls:**
- Detokenization requires `tokens:detokenize` scope
- Every detokenization logged (who, when, reason)
- Rate limit: 100/minute per API key
- Allowed only from whitelisted IPs
