# HSM Reference

Hardware Security Module (HSM) operations for PIN block processing, MAC generation/verification, key management, and EMV cryptographic validation (ARQC/ARPC).

## HSM Overview

Neopay's core engine uses **Thales PayShield 9000/10000** (or GCP Cloud HSM / AWS CloudHSM) for all cryptographic operations. HSMs perform sensitive operations inside the tamper-resistant boundary — keys never exist in the clear outside the HSM.

### Thales PayShield 9000 — Key Commands

```
HOST → HSM: [Command | Key Type | Key Scheme | Encrypted Key | MAC | LRC]
HSM → HOST: [Response Code | Response Data | MAC]
```

**Connection:** TCP/IP on port 8888 (PayShield 9000) or 8443 (TLS)

## Key Hierarchy

```
LMK (Local Master Key) — HSM-resident, never exported
  │
  ├── ZMK (Zone Master Key) — for sending keys to other HSMs
  │     └── ZEK (Zone Encryption Key)
  │     └── ZAK (Zone Authentication Key)
  │
  ├── TMK (Terminal Master Key) — per-terminal master key
  │     ├── TPK (Terminal PIN Key) — encrypts PIN blocks
  │     └── TAK (Terminal Authentication Key) — MAC key
  │
  ├── KEK (Key Encryption Key) — transport key for key delivery
  │     └── DEK (Data Encryption Key) — session encryption
  │
  └── CVK (Card Verification Key) — card authentication
```

## PIN Block Formats (ISO 9564)

| Format | Name | Description | Use |
|--------|------|------------|-----|
| Format 0 | ISO-0 | Encrypted PIN block | Most common |
| Format 1 | ISO-1 | IBM3624 offset | Legacy IBM |
| Format 2 | ISO-2 | Encrypted PIN block + account | MAPCO |
| Format 3 | ISO-3 | IBM RSA PIN block | ECI |
| Format 4 | ISO-4 | Docucentrics / VISA variant | VISA |

### ISO 9564 Format 0 (most common)

```
Byte 1:  0000 0000 (format code = 0)
Byte 2:  [4-bit filler] [12-bit PIN length]
Byte 3-7: [14-digit PIN in BCD]
Byte 8-15: [PAN digits 13-1 or random padding]

PIN "1234", PAN "4111111111111111":
  Block = 0x04 [PIN] [PAN]
```

### PIN Block Translation Command (PayShield)

```
Command: NC (Translate PIN Block)
Key Type: TPK
Input Format: F0 (ISO-0)
Output Format: F0 (ISO-0) or F2 (with account)
Input Key: TPK under LMK
Output Key: TPK under LMK (or new key)

HSM Request:
  NC | TPK-under-LMK | input_pinblock | input_format | 
      output_format | output_key | account_number

HSM Response:
  [Response Code] | output_pinblock
```

## MAC Generation (ISO 9797-1)

MAC algorithms used in payment switch:

| Algorithm | Description | Standard |
|-----------|-------------|----------|
| M1 | DES CBC, zero IV, single length | ISO 9797-1 |
| M2 | DES CBC, zero IV, double length | ISO 9797-1 |
| M3 | DES CBC, ANSI X9.19, double length | ISO 9797-1 (visa variant) |
| M4 | AES CMAC | ISO 9797-1 |

**M1 (ISO 9797-1 Padding Method 1, Block Cipher CBC):**
```java
// Generate MAC using DES M1
// Step 1: Pad data to 8-byte boundary (0x80 + 0x00... + padding length)
// Step 2: XOR first 8 bytes with MAC key
// Step 3: Encrypt XOR result with DES(key, CBC)
// Step 4: XOR next 8 bytes with previous ciphertext
// Step 5: Repeat steps 3-4 for all blocks
// Step 6: Encrypt final result → MAC (8 bytes)

byte[] macKey = hsm.exportKey(KeyType.TAK);
// MAC key encrypted under LMK
byte[] data = buildIso8583Message();
byte[] mac = hsm.generateMAC(data, macKey, Algorithm.M1);
```

## ARQC / ARPC (EMV Cryptography)

### ARQC (Authorization Request Cryptogram)

Generated by the ICC (chip) and verified by the issuer host.

```
ARQC = iAC(PAPDU) using issuer master key
  Input: 5 bytes of ARPC: [ATC(2)] [Unpredictable Number(4)] [Transaction Data]
  iAC:  DES-128-CBC (session key derived from issuer master key + PAN + ATC)
  Output: 8-byte ARQC
```

**Verified by issuer HSM:**
```
Command: KQ (ARQC Verify)
Input:
  - Issuer Master Key (IMK) under LMK
  - PAN (primary account number)
  - ATC (application transaction counter)
  - ARQC (8 bytes from chip)
  - Application cryptogram (AC) received from terminal
  - Unpredictable number (from terminal)
  - Transaction status (CDA, CVM results)

Output:
  - Response: APPROVED / DECLINED / INVALID
  - ARPC (8 bytes) for terminal response
```

### ARPC (Authorization Response Cryptogram)

Generated by issuer, sent back to terminal for card authentication.

```
ARPC = 3DES(ARQC + 2-byte response code + 4-byte issuer auth code)
  Using: Issuer Master Key (IMK) under LMK

Card verification:
  ICC recomputes ARPC from received ARPC input
  Compares with ARPC received from host
  If match → transaction authenticated by issuer
```

### Generate ARPC Command (PayShield)

```
Command: KV (Verify iCac / Generate ARPC)
Input:
  - IMK under LMK (issuer master key)
  - Mode: VERIFY or GENERATE
  - PAN
  - ATC
  - AC (cryptogram from terminal)
  - ARQC
  - Response code (2 bytes)
  - Issuer auth data (4 bytes)

Output:
  - ARPC (8 bytes) if mode = GENERATE
  - Result: APPROVED / DECLINED / INVALID
```

## Key Exchange / Key Loading

### TMK Loading (Host → Terminal)

```
Step 1: HSM generates TMK under ZMK
Step 2: Host sends TMK encrypted under ZMK to terminal
Step 3: Terminal decrypts with ZMK, stores under TMK local
Step 4: Terminal generates working keys (TPK, TAK) under TMK
Step 5: Terminal sends encrypted working keys to host
Step 6: Host verifies MAC on working key package
Step 7: Host stores TPK/TAK under LMK (encrypted)
```

### Key Component Method (Forced Key Entry)

For high-security key injection without HSM-to-terminal transport:

```
Components = 3 (or more) smart cards
  Each card holds a component key (CK1, CK2, CK3)
  TMK = CK1 XOR CK2 XOR CK3
  Components entered by separate key custodians
  TMK never exists in the clear
```

## Key Ceremony (LMK Scheme)

LMK is loaded during **Secure Key Ceremony**:
1. Minimum 3 custodians present
2. Tamper-evident seal verification
3. Hardware audit log opened
4. Components loaded via smart card
5. LMK verification test
6. Ceremony log signed by all custodians
7. Hardware tamper detection active

## Cloud HSM Integration

### AWS CloudHSM

```java
// AWS CloudHSM Java SDK
import com.cloudhsm.model.GenerateRandomRequest;
import com.cloudhsm.model.GenerateRandomResult;

// Generate random for session key
GenerateRandomResult random = cloudHSMClient.generateRandom(32);

// Generate key under CloudHSM
GenerateKeyResult aesKey = cloudHSMClient.generateKey(
    KeyAlgorithm.AES,
    KeyType.SYMMETRIC,
    KeyUsage.ENCRYPT_DECRYPT,
    Extractable.FALSE  // Non-extractable
);

// Encrypt data under CloudHSM
EncryptResult encrypted = cloudHSMClient.encrypt(
    KeyHandle: aesKey.getKeyHandle(),
    PlainText: iso8583MessageBytes,
    Mechanism: Mechanism.AES_CBC_PAD
);
```

### GCP Cloud HSM

```java
// GCP Cloud KMS
CloudKmsClient kms = CloudKmsClient.create();
CryptoKeyName keyName = CryptoKeyName.of("project", "location", "keyring", "key");

// Encrypt PIN block
EncryptRequest req = EncryptRequest.newBuilder()
    .setName(keyName.toString())
    .setPlaintextBytes(ByteString.copyFrom(pinBlock))
    .setPlaintextCoder(PlaintextCodec.UTF8)
    .build();
EncryptResponse resp = kms.encrypt(req);
```

## Key Check Value (KCV)

KCV = first 3 bytes of a key encrypted under a known key:
```
KCV = first_3_bytes( DES(key, "0"*8) )
```

Used to verify key integrity without exposing the key. Terminal and host compare KCVs to confirm matching keys.

## FIPS 140-2 Compliance

Neopay HSM cluster must be **FIPS 140-2 Level 3** validated:
- Tamper-responsive hardware
- Role-based access (crypto officer, cryptouser)
- Multi-factor authentication for key ceremonies
- Firmware integrity verification
- Audit logging of all HSM operations
